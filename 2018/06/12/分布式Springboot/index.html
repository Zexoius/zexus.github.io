微服务+分布式

发表于 2018-06-12 | 分类于 分布式
字数统计: 1,619 | 阅读时长 ≈ 6

最近看微服务与分布式挺火的，自己做一个demo，作为知识储备


这一次尝试Springboot+dubbo+zookeeper+radis。

什么是Springboot？

由其设计目的来说，是为了简化新Spring应用的初始搭建及开发过程。（我们都大概体会过那种痛苦：没有享受大脑风暴流传指尖自由构建小世界的快感，而是模式化重复copy）现在，Springboot提供特定的方式配置，使开发人员不再定义样板化配置。

从字面来说，boot有”启动”之意，所以，Springboot说到底也就是启动Spring项目的工具而已，由根本说来，Springboot就是一些库的集合可以被任意项目构建系统所使用。

Springboot可以做什么？

它不是一个新框架，而是对原有的spring的一个封装，所以，以前spring什么功能，Springboot照样做。

Springboot是一个非常好的微服务框架，用它可以快速搭建一个系统。当然，也可以使用spring cloud（based on Springboot的云应用开发工具）搭建分布式网站。（在电商实战里有具体说明）

什么是zookeeper？

说实话，一开始听到这个我是蒙圈的，”动物管理员”（？？？）竟然有人取这个名字，比我听过Java Android名字来历更雾。好的吧，也不蒙你了，实际上，它就是动物管理员！！

zookeeper用来管理大象（Hadoop）、小蜜蜂（Hive）、小猪（Pig），（？？？怎么那么多动物？不过目前我也只是知道Hadoop是一个大数据框架，其他的咋接触呢，以后再总结吧 (＠_＠ ;) ）

zookeeper，代表着一种技术——分布式协调技术，那么问题来了，什么是分布式协调技术呢？其实它主要用来解决分布式环境中多个进程间的同步控制，使服务器们有序的去访问某种临界资源，防止”脏数据”。有人也许会说“你不能写个调度算法啊！”，一般来说我都不太想解释了，因为他对分布式一定不太了解，所以才会有这种误解。如果，所有的程序都跑在同一台机子上就好了，相对来说这样的确行得通。问题就在于，它是在一个分布式环境下。好的，那么问题又来了，啥是分布式呢？这个我一两句话我也解释不清，不如直接上图。（本人萌新，如有不对轻拍QAQ）

现我有三台机器，每个机器跑一个应用程序。我用网络将其三者连接，构建成一个系统来服务用户。对用户来说这个系统的架构是透明的，他感知不到我这是什么样的架构，此时可以称之为，分布式系统（开发者除外）

那么，现在有一个问题，这个系统如何调用进程？现我在第一个服务器挂载了一个资源，然后这三个物理物理分布的进程都要竞争这个资源，但我们又不希望他们进行同时访问，所以，这里我们需要一个协调器，使得他们能够有序访问这一资源。好，又引出了一个问题，如何实现？目前我只有两种思路即Nginx与ZooKeeper。这两种都可以作为负载均衡服务器，那么他们的区别呢？

Nginx 是著名的反向代理服务器，被广泛作为负载均衡服务器

ZooKeeper 则是分布式协调服务框架，有时候也可作为负载均衡。

Nginx 负载均衡配置

把多个 web server 配置到 Nginx 中，用户访问Nginx时，会自动被分配到某个web server

网站规模变大，通常需要进行服务拆分，每个服务独立部署，通过远程调用方式协调工作。

为了保证稳定性，每个服务不会只使用一台服务器，也会做一个集群存在，那么这个子集群同样需要一个负载均衡器，还是可以使用 Nginx。

到这里，我们发现，根本没有使用 ZooKeeper 的需要嘛，毕竟 Nginx 完全胜任。然而随着你的系统的推进后代的更迭，服务的数量会增加，每个服务集群中的服务器会增加，那么这时候问题就来了。

  1. 维护成本变高了，毕竟你的节点变多了。
  2. 单点故障的风险增加了，因为热点服务的访问量很高，如果这时候此集群服务器的负载均衡器宕机了，那么此服务失效（难以想象(＃°Д°) ）

好，其实都有办法。

第一、自己开发程序解决，但只是降低复杂度，并没有实际解决。

第二、可以通过双机高可用部署方案，使用另一台nginx负载均衡服务器随时待命，只是成本较高。（如果我是老板，我是决计不会这样的，因为这简直是在拔我的腿毛！）

那么，到底咋搞？

此时有人提出了用 ZooKeeper 作为负载均衡的方案。

终于进入了zookeeper说明？（我也是废话够多的。。(￣▽￣)” ）

zookeeper 是一种分布式应用所设计的高可用，高性能切一致的开源协调服务，它提供一项基本服务：分布式锁。老实说，称为锁我不太明白，我在理解消化后，就认为它是一个协调器，所以，其总称在我的脑子里应该是：分布式协调器（emmm这个我前面貌似说过…）。

网络是不稳定的，可能进程繁忙，网络抖动…所以，这就有赖于zookeeper的性能（机制）：数据结构（Znode）+ 原语 + 通知机制（watcher）。这儿就不大仔细分析了，下面直接开整

老样子，还是需要安装客户端，从镜像文件中选取自己所需的版本：有三种可选择。

  1. 单机模式：Zookeeper只运行在一台服务器上，适合测试环境；
  2. 伪集群模式：就是在一台物理机上运行多个Zookeeper 实例；
  3. 集群模式：Zookeeper运行于一个集群上，适合生产环境，这个计算机集群被称为一个“集合体”（ensemble）。

下载完成后，当然是配置环境变量啦，具体教程网上都有资源，自行体验。
